# 实验链与证据链梳理 v3

> 本文档取代 `实验链与证据链梳理v2.md`，作为 ICUS 会议论文实验章节的直接指导文档。v3 的目标不是继续讨论材料是否充足，而是把当前论文故事落实为可写、可跑、可筛选、可控制篇幅的实验链。实验章节服务于 ICUS 论文主线：候选 gap 不是静态选择对象，而是在相邻安全余量约束下以低扰动方式协作发展，并通过扰动-效率联合评分选择发展后的最优 gap。

## 1. 文档定位

ICUS 论文的实验章节采用四段式结构：

```text
IV. Experiments

A. Experimental Setup
   场景、车辆参数、安全距离、评价指标、对比算法和统一比较口径。

B. Feasibility Validation
   展示 candidate gap evaluation -> selected developed gap -> trajectory execution。
   放 candidate score table 和轨迹图。

C. Capacity and Adjustment Ablation
   短实验，合并验证 adjacent-gap capacity 和 space-sensitive QP。
   放一张机制对比图/表。

D. External Algorithm Comparison in Representative Scenarios
   OneStep vs RiosTorres/FIFO。
   当前证据包使用 S01、S02、S03、S06、S08 五个代表性成功合流场景，重点比较主线扰动，而不是把物理入口/出口时间作为无差别性能指标。
```

本文不设置 `E. Discussion`。结论性解释放在各实验小节末尾和 `Conclusion` 中完成。

本文不把实验章写成完整系统报告，不引入随机车流、SUMO 闭环、多 MV rolling、内侧换道辅助 gap supply 或 CAV/HDV 混行。这些内容属于后续论文，不进入 ICUS 实验主线。

## 2. 实验总逻辑

ICUS 实验采用三层证据链：

```text
Feasibility Validation
-> 证明本文方法链可以从候选 gap 评价闭环执行到可行轨迹。

Capacity and Adjustment Ablation
-> 证明相邻 gap 余量约束和空间敏感 QP 的机制行为是合理的。

External Algorithm Comparison
-> 证明在代表性成功合流场景中，OneStep 相比 RiosTorres/FIFO 可以用更少的主线车辆协作完成合流组织；S01 可支撑相近物理合流时间和低扰动，其他场景主要支撑成功合流前提下的低主线扰动。
```

这三层证据分别对应三个审稿人最可能追问的问题：

| 审稿人问题 | 对应实验 | 主文回答 |
|---|---|---|
| 方法链能不能真的从 gap 评价走到轨迹执行？ | B. Feasibility Validation | candidate score table + selected gap execution + MV trajectory |
| adjacent-gap capacity 和 space-sensitive QP 是否只是公式包装？ | C. Capacity and Adjustment Ablation | 四个机制 case 显示算法会按空间余量分配调整，并避免无必要协作 |
| 相比已有 FIFO/RiosTorres 类协调方法，本文方法是否真的低扰动？ | D. External Algorithm Comparison | 先用统一物理入口/出口确认两类方法均完成合流，再比较主线受影响车数、加速度能量和恢复时间 |

## 4. A. Experimental Setup

### 4.1 场景设置

实验采用单帧局部匝道合流决策场景。主线车辆在目标车道上给定纵向位置，MV 根据当前状态评价候选 mainline gaps。每个候选 gap 由后边界车和前边界车定义。

核心设置沿用当前算法文档与 S01 对比报告：

```text
candidate gap length: G_i
required merging gap length: G_req
gap shortage: Delta_i = max(0, G_req - G_i)
front-boundary adjustment: delta_f
rear-boundary adjustment: delta_r
front-side capacity: delta_f_bar
rear-side capacity: delta_r_bar
mainline cooperation cost: C_coop
MV ego merging cost: C_ego
developed-gap score: J_i
```

当前测试用例中 `G_req = 85 m`。相邻安全阈值和车辆动力学约束按代码与实验配置统一设置，论文中只列对实验解释有必要的参数，不展开成系统参数表。

### 4.2 对比算法

外部对比使用：

| 方法 | 含义 |
|---|---|
| OneStep | 本文方法。先评价候选 gap，选择 developed gap，再只对 MV 和选中 gap 的边界车辆生成协作轨迹。 |
| RiosTorres/FIFO | FIFO 队列式协调 baseline。按车辆接近合流区域的顺序生成通过合流区的控制轨迹。 |

内部机制对比使用：

| 方法 | 含义 | 用途 |
|---|---|---|
| Equal split | 将 `Delta_i` 平均分配给前后边界车 | 检查 proposed QP 是否能避免机械平均分配 |
| Unweighted QP | 最小化未加空间权重的二次调整量 | 检查空间敏感权重的作用 |
| Proposed | 使用相邻 gap capacity 和 space-sensitive QP | 本文机制 |

主文 C 节优先使用 `Equal split` 与 `Proposed` 对比。如果表格空间允许，再加入 `Unweighted QP`。`Capacity-proportional` 可作为补充实验或备选 baseline，不作为 ICUS 主文必需项。

### 4.3 统一物理比较口径

外部算法对比不能直接比较各算法内部的 `algorithm_merge_point`，因为 OneStep 的算法合流点是选中 gap 的中心合流位置，而 RiosTorres/FIFO 使用统一物理合流区入口。

需要特别区分两类口径：

```text
1. completion口径：确认两类方法是否都到达统一物理入口和出口；
2. performance口径：比较主线扰动，而不是把入口/出口时间、最小车距和最小TTC作为主性能排序指标。
```

RiosTorres/FIFO 的控制目标是按 FIFO 顺序把车辆组织到固定队列间距中。在本实验实现中，它会围绕固定合流入口和固定 45 m 目标间距调节相关车辆，因此可能通过更大范围的主线车辆加速/减速换取固定间距通行。基于这个机制差异，D 节不应把 `entry time`、`exit time`、`min spacing` 和 `min TTC` 与 OneStep 做逐项优劣比较。它们可以作为完成合流、合流位置设定和安全解释的上下文，但不能作为核心低扰动 claim 的主证据。

外部对比统一使用：

```text
physical merge entry: x = 555 m
physical merge exit:  x = 600 m
```

完成合流确认指标：

```text
MV physical merge entry time
MV physical merge exit time
```

核心主线扰动指标：

```text
affected mainline vehicle count
mainline acceleration energy
mainline speed recovery time
```

规划与安全上下文指标：

```text
OneStep selected-gap merging location p_m
FIFO fixed physical merge entry x = 555 m
minimum spacing
```

其中 `p_m` 与 `minimum spacing` 建议单独列成小型 context table，或在正文中用一句话解释。不要把它们和主线扰动指标混在同一张性能大表中。FIFO 的 `p_m=555m` 和 `min spacing=45m` 是 baseline 设定的一部分，不是与 OneStep 同构的优化结果。

辅助解释指标：

```text
mainline average speed
MV acceleration energy
MV maximum speed
minimum TTC
```

`mainline average speed` 不作为低扰动主张的主指标。若 FIFO 平均速度更高，需要与 `mainline acceleration energy` 和 `affected mainline vehicle count` 同时解释：更高平均速度可能来自更大范围的主线加速，不等于更低扰动。

## 5. B. Feasibility Validation

### 5.1 实验定位

`Feasibility Validation` 是本文实验章的第一层证据。它证明本文方法不是停留在 gap score 或静态 gap selection，而是可以形成完整规划链：

```text
candidate gaps
-> adjacent-gap-preserving cooperation capacity
-> candidate-level minimum adjustment estimate
-> disturbance-efficiency valuation
-> selected developed gap
-> executed boundary adjustment
-> terminal time/location
-> feasible MV trajectory
```

### 5.2 推荐场景

Feasibility Validation 使用 S01 原始基准场景：

```text
X = [-180, -90, -25, 30, 110, 190, 250]
expected best_gap = [30, 110]
expected delta_f = 5
expected delta_r = 0
expected d = 72.5
expected t_m ≈ 13.594
expected p_m ≈ 344.375
```

S01 的作用是展示完整链条，而不是承担所有机制消融和所有外部算法比较。

英文写作句：

> The S01 case demonstrates that the proposed planner forms an executable chain from candidate-gap evaluation and developed-gap selection to terminal condition determination and feasible MV trajectory generation.

## 6. C. Capacity and Adjustment Ablation

### 6.1 实验定位

`Capacity and Adjustment Ablation` 是本文实验章的第二层证据。它合并验证两个机制：

```text
adjacent-gap-preserving cooperation capacity
space-sensitive minimum-disturbance adjustment
```

该实验不追求外部算法性能对比，也不使用“算法无法合流”的失败场景作为主文证据。它只证明本文机制具有以下行为：

```text
1. 当目标 gap 已经满足安全长度时，不产生无意义主线协作；
2. 当前后相邻余量对称时，调整量自然接近对称分配；
3. 当前方相邻余量更大时，优先让前边界车前移；
4. 当后方相邻余量更大时，优先让后边界车后移。
```

### 6.2 四个机制 case

主文 C 节采用四个 case。它们与外部对比候选场景共享，避免额外制造一套不一致实验。

| Case | 对应场景 | 机制现象 | 预期结果 | 主要证明 |
|---|---|---|---|---|
| C1 | S02 | 目标 gap 已满足安全长度 | `Delta=0`, `delta_f=0`, `delta_r=0` | 本文方法不会为了优化评分而无意义扰动主线 |
| C2 | S06 | 前后相邻余量相等 | `delta_f ≈ delta_r` | space-sensitive QP 在对称空间下给出近似对称调整 |
| C3 | S01 | 前方相邻空间更大 | `delta_f > delta_r`, 典型为 `delta_f=5`, `delta_r=0` | 调整负担转向前方空间充足侧 |
| C4 | S08 | 后方相邻空间更大 | `delta_r > delta_f`, 典型为 `delta_f=0`, `delta_r=5` | 调整负担转向后方空间充足侧 |

当前正式证据包不再把 S07 作为独立证据点。前方空间更充足的机制由 S01 的 selected gap 行支撑；论文正文只写机制现象，不写“某场景替代某场景”的数据整理过程。

### 6.3 内部 baseline

C 节 baseline 只服务机制解释，不扩展成大规模消融。

主文保留：

```text
Equal split
Proposed
```

可选保留：

```text
Unweighted QP
```

不在主文展开：

```text
Capacity-proportional
front-only / rear-only
full valuation baseline
parameter sensitivity
```

### 6.4 指标

C 节主文指标控制在一张表内：

| Metric | 含义 | 写作作用 |
|---|---|---|
| `Delta_i` | 当前 gap shortage | 说明需要补足多少 gap |
| `delta_f*`, `delta_r*` | proposed adjustment | 展示调整方向与大小 |
| `delta_f_eq`, `delta_r_eq` | equal split adjustment | 展示朴素平均分配 |
| tight-side margin after adjustment | 紧张侧调整后剩余余量 | 说明 proposed 是否保护紧张侧空间 |
| minimum neighboring-gap margin | 调整后最小相邻安全余量 | 说明没有把风险转嫁给相邻 gap |

`C_coop` 可以作为附加列。若表格过宽，`C_coop` 不进主文，只在正文中用一句话解释 proposed 的低扰动分配来自空间敏感二次代价。

### 6.6 主文结论句

中文逻辑：

> 机制消融表明，本文方法不会在 `Delta=0` 时产生无必要协作；当前后相邻余量对称时，调整近似平均分配；当一侧相邻余量更充足时，space-sensitive QP 会将主要调整量分配给该侧边界车辆，从而减少对紧张侧相邻 gap 的消耗。

英文写作句：

> The ablation cases show that the proposed adjustment model does not introduce unnecessary cooperation when the target gap is already safe, and it shifts the required adjustment toward the boundary vehicle with larger neighboring-gap margin.

## 7. D. External Algorithm Comparison in Representative Scenarios

### 7.1 实验定位

`External Algorithm Comparison` 是本文实验章的第三层证据。它比较 OneStep 与 RiosTorres/FIFO 在代表性成功合流场景中的主线扰动表现。

这里必须先明确 baseline 的机制含义。RiosTorres/FIFO 的目标不是像 OneStep 一样先估计每个 candidate gap 的最小必要协作代价，再选择低扰动 developed gap；它的目标是按 FIFO 顺序组织车辆通过合流区，并使相关车辆追踪固定队列间距。在本实验包中，该固定物理入口为 `x=555 m`，固定间距目标为 `45 m`。因此，FIFO 可能为了形成 45 m 间隔而让多个主线车辆共同加速或减速。这个机制使得 `entry time`、`exit time`、`min spacing` 和 `min TTC` 不适合作为 D 节的主性能比较指标。

该实验的目标是证明：

```text
在两种方法均完成物理合流的代表性场景中，
OneStep 可以减少受影响主线车辆数、主线加速度能量和扰动恢复时间。
其中 S01 还可以支持“相近物理合流时间 + 明显低扰动”的更强表述；
S02/S03/S06/S08 主要支持“成功合流前提下降低主线扰动”，不强行写成所有场景合流时间相近。
```

该实验不承担以下任务：

```text
证明 OneStep 是最快通过合流区的方法；
证明 OneStep 在 entry time、exit time、min spacing、min TTC 上逐项优于 FIFO；
证明 capacity constraint 在合流失败场景中不会强行构造 gap；
证明所有主线密集场景都能被本文方法解决；
证明本文方法已经覆盖 lane-change-assisted gap supply。
```

因此，D 节不纳入 S05 这类 OneStep 可能无法合流的密集不可行场景。合流失败或协作不可行场景可以作为代码测试或后续论文的 gap-supply 问题，不作为 ICUS 外部性能对比主证据。

### 7.2 当前证据包中的代表性场景

当前正式证据包包含五个成功合流场景：

| Scenario | 场景作用 | 预期 OneStep 行为 | 写作功能 |
|---|---|---|---|
| S01 | 原始基准场景 | 选择 `[30,110]`，局部调整前边界车 | 主场景，已有 S01 报告支撑，展示低扰动优势 |
| S02 | 已有安全 gap / 无需协作 | `Delta=0`, 不调整主线 | control case，证明本文方法不会无意义扰动主线 |
| S03 | 可用 gap 在 MV 后方 | `d_i<0`，MV 减速等待后方 gap | 展示方法不只处理前方追赶型合流，也能处理等待型合流 |
| S06 | 前后余量对称 | `delta_f≈delta_r` | 展示外部闭环中的对称协作行为 |
| S08 | rear-side margin larger | `delta_r>delta_f` | 展示后方空间充足时的后边界车优先调整 |

S07 在当前证据包中没有单独运行；前方空间更充足的机制由 S01 的 C 节机制表承担。这个事实只写入实验文档和数据说明，不写入论文正文。论文正文应以机制描述和 Case 编号组织结果，不应出现“某场景替代某场景”之类的数据加工叙述。

这些场景同时覆盖 B、C、D 三个实验小节所需证据：

```text
S01 -> Feasibility Validation + primary external comparison
S02 -> no unnecessary cooperation
S03 -> negative terminal displacement / waiting behavior
S06 -> symmetric adjustment
S01 -> front-side preference in the capacity-and-adjustment ablation
S08 -> rear-side preference
```

### 7.3 正文场景选择规则

写论文时正文不必展示所有场景。D 节建议把正文场景抽象为 Case A-E，避免用 S01/S02 这类运行编号承担学术叙事。内部数据表和证据文件保留 scenario id 以便追踪即可。

正文展示场景按以下规则选择：

```text
1. 用统一物理入口/出口确认 OneStep 与 FIFO 都完成合流。
2. D 节主表优先展示所有五个有效场景的主线扰动指标，场景名可写为 Case A-E。
3. D 节正文解释不逐个堆叠场景，而按现象归纳：S01/Case A 支撑相近合流时间和低扰动；S02/S03 支撑无需主线协作时的零扰动；S06/S08 支撑需要协作时仍能缩小主线扰动范围和恢复时间。
4. 如果篇幅不足，可保留主表全部行，但正文只解释最有代表性的 2-3 类现象。
5. 最终选择不能只按结果最漂亮，而要按“数据有效 + 机制覆盖 + 低扰动结论清楚”三条标准。
```

推荐正文写法：

```text
主表：Case A-E 全部列出 disturbance metrics。
正文：重点解释 Case A、零扰动 cases、以及需要协作的 directional/symmetric cases。
图：最多保留一张 time-space comparison，用于说明 FIFO 固定间距队列组织与 OneStep selected-gap execution 的机制差异。
```

最终组合以真实数据质量为准。正文只写能清楚支撑低扰动主线的现象；不把异常、重复或解释负担过大的细节硬塞进主文。

### 7.4 外部对比指标

D 节主表只使用主线扰动指标：

| 指标 | OneStep 预期优势 | 解释 |
|---|---|---|
| `affected_mainline_vehicle_count` | 更低 | 说明扰动范围更小 |
| `mainline_acceleration_energy` | 更低 | 说明主线控制强度更小 |
| `mainline_speed_recovery_time` | 更短 | 说明扰动持续时间更短 |

以下指标不进入主比较表：

| 指标 | 处理方式 |
|---|---|
| `physical_merge_entry_time` | 只用于确认完成合流，S01 可用来说明时间相近；不能对所有场景写成本文方法与 FIFO 合流时间相近 |
| `physical_merge_exit_time` | 同上 |
| `min_spacing` | 可与 `p_m` 一起作为 planning/safety context 单独列小表或正文说明 |
| `min_ttc` | 仅作为辅助安全解释；不作为 D 节主性能比较指标 |
| `mainline_average_speed` | FIFO 可能更高，但可能来自更大范围主线加速，不等于低扰动 |

辅助指标：

| 指标 | 解释方式 |
|---|---|

| `MV maximum speed` | 用于解释 MV 自车代价，不作为本文主线扰动 claim 的主证据 |

S01/Case A 可以单独写一句更强表述：

> In the primary representative case, both methods complete the physical merge with nearly identical entry and exit times. ### 7.7 主文结论句

中文逻辑：

> 在代表性成功合流场景中，OneStep 与 RiosTorres/FIFO 均能完成物理合流。与 FIFO 相比，OneStep 的主要优势不是更早到达固定物理入口，也不是在固定 45 m 间距指标上与 FIFO 排名比较，而是通过 candidate-level developed-gap valuation 和更局部的边界车辆协作减少受影响主线车辆数、主线加速度能量和扰动恢复时间。S01 可进一步说明在相近物理合流时间下仍能显著降低主线扰动；其他场景则支撑成功合流前提下的低扰动特性，部分场景可能以 MV 物理进入/离开延后为代价。

英文写作句：

> Across representative successful-merging cases, the proposed method reduces the spatial extent, control intensity, and recovery duration of mainline disturbance relative to the FIFO baseline. In the primary case, this reduction is achieved with nearly identical physical merge timing; in the other cases, the lower disturbance may be obtained at the cost of later MV physical entry or exit.

## 10. 与 Method 和 Introduction 的对齐

### 10.1 Method 对齐

Method 中各模块与实验对应如下：

| Method 模块 | 对应实验 | 写作边界 |
|---|---|---|
| Problem Formulation | B + D | 说明主线协作有代价，不能写成全局交通最优问题 |
| Adjacent-Gap-Preserving Cooperation Capacity | C | 强调相邻余量决定可调整方向与容量 |
| Space-Sensitive Minimum-Disturbance Adjustment | C | 强调调整负担随空间余量自动分配 |
| Disturbance-Efficiency Valuation | B | 用 candidate score table 展示评分与选择 |
| Terminal Time/Location and Trajectory | B | 用 S01 轨迹图展示可执行性 |
| Overall Low-Disturbance Effect | D | 用 OneStep vs FIFO 的主线扰动指标支撑外部效果，避免把固定入口时间或固定间距当作主性能比较 |

Method 中不得把 C 节写成“完整系统消融已经证明所有模块最优”。正确写法是：

```text
The ablation cases verify the intended behavior of the proposed capacity and adjustment mechanisms.
```

### 10.2 Introduction 对齐

Introduction 中的实验钩子改成三层：

```text
First, a feasibility case demonstrates that the proposed chain can evaluate candidate gaps, select the best developed gap, and generate an executable merging trajectory.

Second, compact ablation cases verify that the proposed capacity and adjustment mechanisms allocate cooperation according to neighboring-gap margins.

Third, representative comparisons with a FIFO baseline show that the proposed method reduces mainline disturbance in successful-merging cases; the primary case additionally shows that this can be achieved with nearly identical physical merge timing.
```

Introduction 不写：

```text
large-scale statistical validation
random traffic validation
SUMO closed-loop validation
mixed CAV/HDV validation
```

## 11. 当前版本最终判断

v3 的实验链是 ICUS 论文的主文实验方案。

最终实验章由以下三类证据组成：

```text
1. S01 Feasibility Validation
2. Four-case Capacity and Adjustment Ablation using S02/S06/S01/S08
3. OneStep vs RiosTorres/FIFO comparison over S01/S02/S03/S06/S08 representative successful-merging cases
```

写论文时，D 节不需要把所有运行编号都作为叙事对象。主文应围绕主线扰动指标组织结果，可把场景写成 Case A-E；内部证据文件保留 S01/S02/S03/S06/S08 以便追踪。S01/Case A 是主场景；重复场景、解释负担过重场景和无法完成合流的场景不进入 D 节主文。

这套设计能够支撑 ICUS 论文的核心论断：

> 本文提出的 mainline-disturbance-aware cooperative gap development 方法可以把候选 gap 评价、空间余量约束、最小扰动协作调整、扰动-效率联合评分和可行轨迹生成连接成一条可执行链，并在代表性成功合流场景中相较 FIFO 类协调方法减少主线协作范围、主线加速度能量和扰动恢复时间。
