# 拆分指导思路

## 一、两篇论文的拆分版本

## 论文 1：ICUS 会议论文

### 1. 论文定位

**主题定位：**

> 面向匝道合流的主线扰动感知协作 gap development 方法。

更具体地说，ICUS 论文关注的是：

> 在合流交互区域内，给定若干候选主线 gap，如何把主线车辆协作看作一种有边界、有代价的局部交通资源，通过前后边界车辆的纵向协作，以较小主线扰动发展并选择一个可执行的合流 gap。

这篇论文的主角不是“可达集”，也不是“长匝道全局搜索”，而是：

```text
局部候选 gap
    ↓
主线协作能力有边界
    ↓
gap 缺口需要低扰动分配
    ↓
发展后的 gap 需要评分选择
    ↓
选中 gap 需要转化为合流时空点和可行轨迹
```

原始文档中已经把 gap development 理解为“有约束、有代价的局部交通资源分配问题”，即 gap 不是静态选择对象，而是通过有限主线协作发展出来，并通过主线协作代价和 MV 自车代价联合评价。这个核心思想保留为 ICUS 论文的中心。fileciteturn14file14

---

### 2. 核心问题

ICUS 论文回答的问题是：

> 在局部合流交互区域内，如何对若干候选 gap 进行主线扰动感知的协作发展、低扰动调整、综合评分与轨迹执行？

更凝练地写：

> 如何将候选 gap 的发展过程建模为一种有限、带代价的主线协作资源分配问题，并在发展后选择综合代价更优、轨迹可执行的合流机会？

这里不再引入“可达集”问题。第一篇不讨论“从长匝道全局车辆流中筛选哪些 gap 可达”，而是从局部交互区内的候选 gap 发展与评价切入。



```text

```

并且不会把论文说成一个大系统。

### 4. ICUS 论文贡献点

先不要吞贡献点，建议保留 4 条贡献，但正文中可以根据篇幅合并成 3 条。

#### 贡献 1：主线扰动感知的协作 gap development 问题定义

提出一种主线扰动感知的协作 gap development 问题表述。与把主线车辆视为可自由调度资源的全局优化方法不同，本文把当前合流所需的边界主线车辆协作视为一种**有限、有代价、需要最小化扰动的局部交通资源**。

建议中文贡献表述：

> 提出一种主线扰动感知的协作 gap development 问题表述，将候选 gap 的发展过程建模为由前后边界车辆参与的有限主线协作资源分配问题，并显式考虑主线协作扰动代价。

对应原始创新点：

```text
贡献点 2：主线扰动感知的协作 gap development 问题定义
```

原有文档中已经明确建议，不要把主线车辆写成免费协调对象，而应强调只控制当前 MV 合流所必要的边界 CV，并显式惩罚协作调整量。fileciteturn14file12

---

#### 贡献 2：基于相邻 gap 余量的协作能力约束模型

提出基于相邻 gap 余量的协作能力模型，用候选 gap 前后相邻空间刻画前后边界车辆最大可调整能力。

核心思想：

```text
前边界车能向前调整多少，取决于它前方相邻 gap 还能释放多少安全余量；
后边界车能向后调整多少，取决于它后方相邻 gap 还能释放多少安全余量；
当前 gap 的可扩张能力不是固定的，而是由周边主线交通空间共同决定。
```

建议中文贡献表述：

> 提出一种基于相邻 gap 余量的局部协作能力约束模型，用前后相邻 gap 的剩余安全空间刻画候选 gap 边界车辆的最大可调整量，避免当前 gap 的扩张以压缩相邻 gap 安全余量为代价。

对应原始创新点：

```text
贡献点 3：基于相邻 gap 余量的协作能力约束模型
```

这一点在原始文档中已经被定义为“不是普通安全距离检查”，而是将当前 gap development 与相邻主线交通空间耦合起来。fileciteturn14file12

---

#### 贡献 3：空间敏感的最小扰动协作调整优化

构造空间敏感的最小扰动协作调整优化问题，根据前后边界车辆的可调整空间自动分配 gap 缺口。

核心思想：

```text
不是平均分配；
不是只让前车承担；
不是只让后车承担；
不是固定比例分配；
而是让空间更充足的一侧承担更多调整，让空间紧张侧少受扰动。
```

建议中文贡献表述：

> 构造一种空间敏感的最小扰动协作调整优化方法，根据前后边界车辆的局部可用空间自动分配 gap 缺口，在满足安全 gap 需求的同时降低主线协作扰动和单车峰值调整量。

对应原始创新点：

```text
贡献点 4：空间敏感的最小扰动协作调整优化
```

这一贡献是 ICUS 论文最核心的技术模块之一。已有拆分建议中也指出，空间敏感 QP 的价值不是“用了一个 QP”，而是在同样补足 gap 缺口的条件下，减少空间紧张侧和最大单车扰动。fileciteturn14file1

---

#### 贡献 4：扰动-效率联合的 gap 评分与选择机制

提出发展后 gap 的扰动-效率联合评分机制，将主线协作代价和 MV 自车合流代价统一到同一评价函数中。

核心思想：

```text
最大 gap 不一定最好；
最近 gap 不一定最好；
MV 自车代价最低的 gap 不一定最好；
主线协作代价最低的 gap 也不一定最好；
真正应该比较的是“协作发展后的综合代价”。
```

建议中文贡献表述：

> 提出一种协作发展后的扰动-效率联合 gap 评分机制，将最小主线协作代价与 MV 自车合流代价统一到同一评价函数中，从而在多个发展后的候选 gap 中选择综合代价更优的合流机会。

对应原始创新点：

```text
贡献点 5：扰动-效率联合的 gap 评分与选择机制
```

这一点应在 ICUS 中保留为正式贡献，因为它把“gap 能不能被发展”进一步闭环到“多个发展后的 gap 该选哪个”。原始实验链中也将 gap valuation 实验作为证明“最优 gap 不一定最近、最大或 ego 最省力”的关键证据。fileciteturn14file15

---

#### 贡献 5：灵活合流时空点与闭式可行轨迹生成

这一点可以保留，但在论文贡献列表中可以与贡献 4 合并。

核心思想：

```text
合流时间不是预设的；
合流位置不是预设的；
轨迹终端不是独立指定的；
它们由选中的 gap 及其协作发展结果共同决定。
```

建议中文贡献表述：

> 基于协作后 gap 中心确定 MV 的灵活合流时间和合流位置，并生成满足速度、加速度和终端连续性约束的闭式可行轨迹，实现从 gap 发展、gap 选择到轨迹执行的完整闭环。

对应原始创新点：

```text
贡献点 6：灵活合流时空点与闭式可行轨迹生成
```

这里不应把“五次轨迹”本身包装成大创新。真正值得强调的是：合流时间、合流位置和轨迹由协作后 gap 中心驱动，而不是人为预设。fileciteturn14file7

### 5. ICUS 论文贡献写法版本

如果写成 4 条：

```text
1. 提出一种主线扰动感知的协作 gap development 问题表述，将主线边界车辆协作建模为有限、有代价的局部交通资源。

2. 提出基于相邻 gap 余量的协作能力约束模型，刻画候选 gap 前后边界车辆在不压缩相邻安全余量条件下的最大可调整能力。

3. 构造空间敏感的最小扰动协作调整优化方法，根据局部可用空间自动分配 gap 缺口，降低主线协作扰动和单车峰值调整量。

4. 提出协作发展后的扰动-效率联合 gap 评分与选择机制，并基于协作后 gap 中心确定灵活合流时空点和闭式可行轨迹。
```

如果写成 3 条：

```text
1. 提出主线扰动感知的协作 gap development 问题表述，将候选 gap 的发展建模为有限、有代价的局部主线协作资源分配问题。

2. 提出基于相邻 gap 余量的协作能力约束模型，并构造空间敏感的最小扰动协作调整优化方法，实现安全余量保持下的低扰动 gap 扩张。

3. 提出协作发展后的扰动-效率联合 gap 评分与选择机制，并由协作后 gap 中心驱动灵活合流时间、合流位置和闭式可行轨迹生成。
```

ICUS 正文更推荐 3 条，因为贡献表述更紧凑，但内部写作时仍按 5 个模块组织。

### 6. ICUS 论文方法结构

建议结构：

```text
I. Introduction

II. Problem Formulation
   A. Local cooperative gap development scenario
   B. Candidate gap, boundary vehicles, and safety margin
   C. Mainline disturbance and ego merging cost

III. Proposed Method
   A. Adjacent-Gap-Based Cooperation Capacity
   B. Space-Sensitive Minimum-Disturbance Adjustment
   C. Disturbance-Efficiency Gap Valuation
   D. Flexible Terminal Time/Location and Trajectory Generation

IV. Simulation Results
   A. Feasibility validation
   B. Adjacent-gap capacity comparison
   C. Space-sensitive adjustment ablation
   D. Gap valuation comparison

V. Conclusion
```

注意：这里不出现可达集，不出现 reachability-aware candidate construction，不出现全局候选 gap 筛选。

### 7. ICUS 论文实验设计

ICUS 实验应围绕机制证据链组织，而不是大规模系统报告。

#### 实验 1：完整可行性验证

目的：

```text
证明方法能从候选 gap 发展、边界车调整、gap 评分、合流时空点确定一直闭环到可行轨迹生成。
```

建议展示：

```text
主线车辆位置演化
gap 长度变化
前后边界车辆调整量
MV 轨迹
MV 速度与加速度
合流时间
合流位置
最小安全间距
```

结论口径：

> 所提方法能够在给定合流交互场景下生成满足安全、速度、加速度约束的可执行合流轨迹。

---

#### 实验 2：相邻 gap 余量协作能力实验

对比方法：

```text
无相邻 gap 约束
固定最大调整量
本文相邻 gap 余量约束
```

证明对象：

```text
主线协作能力不是固定资源；
gap 扩张不能以牺牲相邻 gap 安全余量为代价；
本文方法能够保留相邻 gap 的安全空间。
```

---

#### 实验 3：空间敏感 QP 消融实验

对比方法：

```text
Equal split
Front-only
Rear-only
Capacity-proportional
Unweighted QP
Proposed space-sensitive QP
```

核心指标：

```text
紧张侧调整量
最大单车调整量
相邻 gap 剩余余量
协作车辆速度偏差
协作车辆加速度能量
主线协作代价 C_coop
```

原始实验链中已经指出，QP 的优势不在于“是否补足 gap 缺口”，而是在相同 gap development 需求下，把协作负担更多分配给空间充足侧，减少空间紧张侧和单车峰值扰动。fileciteturn14file5

---

#### 实验 4：扰动-效率 gap valuation 实验

对比方法：

```text
Nearest gap
Largest gap
Minimum ego cost
Minimum cooperation cost
Existing-safe-gap only
Proposed joint valuation
```

证明对象：

```text
最优 gap 不一定最近；
不一定最大；
不一定 MV 自车代价最低；
不一定主线协作代价最低；
而应是协作发展后主线扰动与 MV 合流效率综合更合理的 gap。
```

推荐图表：

```text
二维 trade-off 图：
横轴：MV 合流效率，例如 C_ego 或 merge time
纵轴：主线扰动，例如 C_coop、最大单车调整量、速度损失
```

### 8. ICUS 写作口径

ICUS 论文不主动强调：

```text
我们没有处理长匝道；
我们没有做 rolling；
我们没有处理多 MV；
我们不能处理高密主线；
我们没有 SUMO；
我们没有 HDV 混行。
```

这些不应在 Introduction 中被放大。

更好的写法是自然限定：

```text
本文研究合流交互区域内的协作 gap development 问题。
本文关注候选 gap 的协作发展、主线扰动分配、发展后评分和轨迹执行。
```

在方法和实验中自然呈现：

```text
候选 gap 数量有限；
主线边界车纵向协作；
数值仿真验证；
单次合流决策。
```

Conclusion 中可以轻描淡写地写：

> 后续将进一步扩展到更长时空范围下的 rolling coordination 和更复杂交通流验证。

不要写成：

> 本文无法处理……

而应写成：

> 后续将扩展……

这能保护第一篇的完整性和审稿观感。

## 论文 2：后续论文

### 1. 论文定位

后续论文不是“ICUS + 一个换道分支 + SUMO 实验”，而是问题域升级：

> 从局部 gap development 升级为长匝道、多车、rolling 条件下的 hybrid gap supply 与资源预约重规划问题。

第二篇的主角是：

```text
长匝道区域
多 MV rolling
大量主线车辆与候选 gap
密集主线下纵向发展不足
可达 action set
纵向调整 / 等待 / 主线车向内侧换道
多 MV 资源预约与重规划
随机车流与 SUMO 闭环验证
可选：CAV/HDV 混行鲁棒性
```

已有拆分建议中也指出，单独“增加内侧换道分支”或“从数值仿真升级到 SUMO”都不足以支撑新论文；但它们与多 MV rolling 协调结合后，可以形成“高密主线中多个匝道车如何在有限、可竞争、跨车道耦合的主线资源下动态供应并分配合流机会”的新问题。fileciteturn14file0

---

### 2. 核心问题

后续论文回答的问题是：

> 当问题从局部合流交互区扩展到长匝道、多 MV、rolling planning 和密集主线交通时，如何构造可执行的 gap-supply action set，并在纵向调整、等待、主线车向内侧换道等多种模式之间进行选择、预约和重规划？

更凝练地写：

> 如何在长时空 rolling 场景中，为多个 MV 动态供应、分配和重规划合流机会？

这篇论文的“可达集”不再是第一篇的缺失模块，而是第二篇问题域自然产生的新模块。

### 4. 后续论文贡献点

#### 贡献 1：rolling 场景下的可达 gap/action set 构造

第二篇中的“可达集”不应只是简单筛 gap，而应升级为 rolling horizon 下的**可达 action set**。

第一篇处理的是：

```text
局部候选 gap 的 development 与 valuation
```

第二篇处理的是：

```text
长匝道 rolling 场景中，哪些 gap / 哪些模式 / 哪些时间窗 / 哪些协作车辆组合构成可执行 action
```

建议中文贡献表述：

> 提出一种面向长匝道 rolling 场景的可达 gap/action set 构造方法，根据 MV 运动学约束、时间窗、候选 gap 时空位置、协作模式可行性和已有资源预约，筛选每个 MV 在当前 rolling horizon 内可执行的 gap-supply action。

这里的 action 可以定义为：

```text
a = {
    MV 编号,
    目标 gap,
    gap supply 模式,
    协作车辆,
    目标合流时间窗,
    资源预约区间,
    预期扰动代价
}
```

这比“可达集筛选”更强，也能自然支撑第二篇的 rolling 与多模式选择。

---

#### 贡献 2：从 gap development 到 hybrid gap supply

第二篇把第一篇的纵向 gap development 扩展为多模式 gap supply。

可包含模式：

```text
Mode 1：纵向 gap development
    继承第一篇方法，作为 longitudinal branch。

Mode 2：减速等待 / defer
    当当前 gap supply 代价高或不可行时，让 MV 等待后续窗口。

Mode 3：主线车向内侧换道释放空间
    当目标主线纵向余量不足时，通过内侧车道接收与主线车换道释放目标车道空间。

Mode 4：组合模式
    例如轻微纵向调整 + 换道释放 + rolling 重规划。
```

建议中文贡献表述：

> 提出一种 hybrid gap supply 框架，将局部纵向 gap development 扩展为包含纵向调整、减速等待和主线车向内侧换道释放空间的多模式合流机会供应机制，以应对密集主线中单纯纵向调整能力不足的场景。

---

#### 贡献 3：mode-aware 评价与选择机制

第二篇不一定要用一个大优化问题吃掉整个状态机，但必须有跨模式可比较的评价原则。

建议采用：

```text
分模式求解，统一评价。
```

即：

```text
纵向调整模式：调用第一篇 longitudinal gap development planner；
等待模式：计算等待代价、延误代价和后续 gap 可用性；
换道辅助模式：计算内侧车道接收可行性、换道扰动、目标车道释放收益；
最终在可行 action set 内进行 mode-aware valuation。
```

建议中文贡献表述：

> 构造一种 mode-aware gap-supply 评价机制，对纵向调整、等待和换道辅助等不同模式下的候选 action 进行统一比较，综合考虑 MV 合流效率、主线纵向扰动、换道扰动、内侧车道影响和资源占用代价。

这一贡献与第一篇的 gap valuation 有继承关系，但不重合：

```text
第一篇 valuation：
    比较局部候选 gap 在纵向协作发展后的综合代价。

第二篇 valuation：
    比较 rolling 场景中的 gap + mode + time window + reservation action。
```

---

#### 贡献 4：多 MV rolling 资源预约与重规划

第二篇需要把多 MV rolling 变成方法贡献，而不是单纯实验。

```text
可达性筛选

reachability-aware

先判断 MV 是否能到达 gap
```

```text
gap development 消耗主线车辆速度、位置和相邻空间余量；
```

开始讲。

## 3. 第一篇如何避免暴露“场景太局部”的弱点

第一篇不需要反复强调：

```text
单 MV；
很局部；
不 rolling；
不高密；
不换道；
不 SUMO。
```

这些表达会把论文弱点主动放大。

更好的表达是：

```text
本文研究合流交互区域内的 cooperative gap development。
本文关注候选 gap 的协作发展、主线扰动分配、发展后评价和轨迹执行。
本文提供一种轻量、可解释、可闭式执行的局部规划框架。
```

也就是说，论文中可以“自然呈现”场景设定，但不要“防御式声明”自己没做什么。

建议使用的表达：

```text
合流交互区域
候选 gap
边界主线车辆
局部主线协作
主线扰动感知
协作发展后的 gap 评价
可执行合流轨迹
```

避免使用的表达：

```text
只适用于很小局部场景
不处理长匝道
不处理不可达 gap
不处理复杂交通
高密主线会失败
```

## 4. 第二篇为什么自然需要可达集

第二篇的问题域变了。

在第一篇里，候选 gap 数量有限，且都处在合流交互区域内；此时研究重点是：

```text
一个候选 gap 如何发展；
多个发展后 gap 如何评分；
选中 gap 如何执行。
```

在第二篇里，场景变成：

```text
长匝道；
多 MV；
大量主线车辆；
多个未来时间窗；
多个 gap supply 模式；
rolling horizon；
资源预约和重规划。
```

此时如果仍然对所有 gap、所有模式、所有时间窗都做发展与评分，会产生大量无意义计算，也可能生成实际不可执行的协作请求。因此第二篇引入可达集是自然的，而且应升级为：

```text
可达 gap/action set
```

这让第二篇不是“补上第一篇没做的约束”，而是：

```text
从局部候选 gap evaluation
升级到
长时空 rolling action generation。
```

这是两篇论文之间最关键的逻辑桥。


## 7. HDV/CAV 混行的处理

HDV/CAV 混行可以作为第二篇增强方向，但暂时不建议把它放到最核心位置。

推荐层级：

```text
第一层核心：
    reachable action set
    hybrid gap supply
    rolling reservation and replanning

第二层验证：
    random traffic
    SUMO closed-loop

第三层增强：
    CAV/HDV mixed traffic
    HDV compliance probability
    different CAV penetration rates
```

如果混行只是通过 SUMO 中加入 HDV 并观察算法表现，它应作为实验鲁棒性验证。

如果后续将 HDV 是否服从、HDV gap acceptance、HDV 预测不确定性写入 action feasibility 或 expected cost，则可以提升为正式贡献。
