**一、这次到底跑出了什么数据**

这次 S01 实验一共跑出了两套算法结果：

| 算法 | 含义 |
|---|---|
| `one_step` | 我们项目的单帧 OneStep 算法，先选 gap，再只对 MV 和选中 gap 边界车生成协作轨迹 |
| `rios_torres_fifo` | RiosTorres/FIFO baseline，按“谁离合流区入口更近谁先过”的 FIFO 队列，为队列车辆生成最优控制轨迹 |

原始轨迹表 `comparison_trajectory.csv` 有 `6192` 行。它是后续所有指标和图的基础数据。里面每一行是一辆车在某个时刻的状态，关键列包括：

```text
algorithm, scenario_id, vehicle_id, role, road, queue_order,
t, x, v, a, x0, v0,
is_mv, is_mainline, is_controlled,
algorithm_merge_time, algorithm_merge_point,
physical_merge_entry_time, physical_merge_entry_x,
physical_merge_exit_time, physical_merge_exit_x
```

这里要特别区分：

| 字段 | 类型 | 含义 |
|---|---|---|
| `t, x, v, a` | 原始轨迹采样数据 | 某车在某时刻的位置、速度、加速度 |
| `x0, v0` | 原始初始状态数据 | 该车初始位置、初始速度 |
| `is_controlled` | 原始/标记数据 | 该车是否被算法主动规划/控制 |
| `algorithm_merge_time/point` | 算法结果数据 | 各算法自身定义下的“合流时间/合流点” |
| `physical_merge_entry/exit` | 统一物理口径数据 | 统一用 `x=555m` 作为合流区入口，`x=600m` 作为出口 |

`summary.json` 给的是实验摘要和算法关键决策。S01 的核心摘要是：

| 项目 | OneStep | RiosTorres FIFO |
|---|---:|---:|
| 选中/通过关系 | `best_gap = gap4 = [30, 110]` | FIFO 队列固定 |
| 算法合流时间 | `13.59375s` | `24.25s` |
| 算法合流点 | `344.375m` | `555.0m` |
| 物理合流入口时间 | `24.125s` | `24.25s` |
| 物理合流出口时间 | `26.375s` | `26.5s` |

这里最重要的一点是：`algorithm_merge_point` 不能直接粗暴比较。OneStep 的 `344.375m` 是“算法选择的 gap 中心合流位置”，RiosTorres 的 `555m` 是“统一物理合流区入口 L”。真正同一物理位置下的效率比较，要看 `physical_merge_entry_time` 和 `physical_merge_exit_time`。

RiosTorres FIFO 的真实队列是：

```text
target_lane_250m
target_lane_190m
target_lane_110m
target_lane_30m
merge_vehicle
target_lane_-25m
target_lane_-90m
target_lane_-180m
```

它说明 baseline 没有像 OneStep 一样主动挑 `[30,110]` 这个 gap，而是严格按照距离合流入口的前后顺序排队。

**三、metrics.csv 里的指标怎么来的**

下面这些都是加工后的计算指标，不是原始采样值。公式来自当前 `metrics.py` 的实际计算口径。

| 指标 | 计算公式/口径 |
|---|---|
| `mv_completed` | MV 是否有 `physical_merge_exit_time` |
| `mv_algorithm_merge_time` | MV 到达算法自身合流点的时间 |
| `mv_algorithm_merge_point` | 算法自身合流点 |
| `mv_physical_merge_entry_time` | MV 首次到达统一物理入口 `x=555` 的时间 |
| `mv_physical_merge_exit_time` | MV 首次到达统一物理出口 `x=600` 的时间 |
| `mv_merge_speed_at_algorithm_point` | 对 MV 轨迹按 `algorithm_merge_time` 插值得到速度 |
| `mainline_average_speed` | 所有主线车辆所有采样点 `v` 的平均值 |
| `mainline_min_speed` | 所有主线车辆采样点的最小 `v` |
| `mainline_max_speed_drop` | `max(max(0, 20-v))` |
| `mainline_speed_recovery_time` | 最后一个满足 `abs(v-20)>0.1` 或 `abs(a)>0.05` 的主线采样时刻 |
| `mainline_total_delay` | 对每辆主线车积分：`sum(max(0, 20-v) * dt)` |
| `mainline_acceleration_energy` | 对每辆主线车积分：`sum(a^2 * dt)` |
| `max_single_vehicle_deceleration` | 当前实现是所有车辆中的最小 `a`，包含 MV，不是主线专属 |
| `affected_mainline_vehicle_count` | 主线车中 `max(abs(v-20))>0.1` 或 `max(abs(a))>0.05` 的车辆数 |
| `hard_brake_or_stop` | 是否 `min(a)<=-3.0` 或 `min(v)<=0.5` |
| `min_spacing` | 同一路、同一时刻相邻车辆 `x_front-x_rear` 的最小值 |
| `min_ttc` | 若后车比前车快，`TTC=spacing/(v_rear-v_front)`，取最小值 |

**六、我们的算法在哪些数据上更好**

OneStep 明显更好的指标是主线扰动类、安全裕度类、传播范围类：

1. 主线受影响车辆数：`1` vs `6`
   - OneStep 只调整选中 gap 的前车。
   - FIFO 影响了除最前车外的大部分主线车。

2. 主线加速度能量：`0.1706` vs `22.1881`
   - 公式：`sum(a^2 * dt)`
   - FIFO 是 OneStep 的约 `130` 倍。
   - 这说明 FIFO 虽然速度更高，但控制动作强度大得多。

3. 主线速度恢复时间：`13.1s` vs `31.0s`
   - 公式：最后一个满足 `abs(v-20)>0.1` 或 `abs(a)>0.05` 的主线时刻。
   - OneStep 的扰动在约 `13.1s` 后结束。
   - FIFO 的主线扰动持续到 `31.0s`，几乎贯穿整段队列调度。

4. 最小车间距：`55.0m` vs `45.0m`
   - FIFO 正好压到设定的同路安全距离 `45m`。
   - OneStep 的最小间距还有 `55m`，安全裕度更大。

5. 最小 TTC：`112.32s` vs `22.31s`
   - 两者都不危险，但 OneStep 的 closing 风险明显更低。
   - FIFO 的最小 TTC 发生在 `t=19.75s`，`target_lane_30m` 追近 `target_lane_110m`，spacing 约 `47.49m`，后车速度 `22.13m/s`，前车 `20m/s`。

6. 物理合流效率没有变差
   - OneStep 到 `x=555` 是 `24.125s`，FIFO 是 `24.25s`。
   - OneStep 到 `x=600` 是 `26.375s`，FIFO 是 `26.5s`。
   - 也就是说 OneStep 在主线扰动小很多的情况下，物理合流还略快 `0.125s`。

**七、FIFO 哪些数据上更好**

FIFO 也不是没有优点，它在下面几个指标上表现更好或看起来更好：

1. 主线平均速度更高：`21.4289m/s` vs `20.0186m/s`
   - 这是 FIFO 最明显的“效率型优势”。
   - 但要解释清楚：这个优势来自让多辆主线车加速，不是来自更小扰动。
   - 如果论文目标是“提高主线平均速度”，FIFO 这项更好；如果目标是“减少主线扰动”，这项不能单独当作优点。

2. MV 自身轨迹更温和
   - OneStep MV 最大速度约 `30.0m/s`，最小加速度约 `-2.265m/s²`。
   - FIFO MV 最大速度约 `24.33m/s`，最小加速度约 `-0.714m/s²`。
   - MV 加速度能量：OneStep 约 `35.87`，FIFO 约 `4.15`。
   - 所以如果单看 MV 乘坐舒适性/控制强度，FIFO 更平滑。

3. 主线最小速度、主线速度下降、主线总延误三项持平
   - 两边主线最小速度都是 `20m/s`。
   - 两边速度下降最大值都是 `0`。
   - 两边总延误都是 `0`。
   - 这说明 S01 这个场景里，两套算法都没有让主线车低于参考速度；区别主要在“谁加速了多少、影响了几辆车、持续多久”。

**九、这组 S01 数据可以支持的结论**

比较稳的结论可以这样写：

> 在相同 S01 单帧场景下，OneStep 与 RiosTorres FIFO 都能使 MV 完成物理合流，且到达统一合流入口/出口的时间几乎相同。OneStep 的物理入口时间为 `24.125s`，FIFO 为 `24.25s`；OneStep 的物理出口时间为 `26.375s`，FIFO 为 `26.5s`。因此，两者合流效率基本相当，OneStep 还略快。

> 但在主线扰动方面，OneStep 明显更小。OneStep 只影响 `1` 辆主线车，主线加速度能量为 `0.1706`，扰动恢复时间为 `13.1s`；RiosTorres FIFO 影响 `6` 辆主线车，主线加速度能量为 `22.1881`，扰动恢复时间为 `31.0s`。这说明 OneStep 在保持合理合流效率的同时，显著减少了主线协作扰动。

> FIFO 的主线平均速度更高，`21.4289m/s` 对比 OneStep 的 `20.0186m/s`，但这是通过更大范围的主线车辆加速实现的。因此如果评价目标是“速度均值”，FIFO 占优；如果评价目标是“低扰动合流”，OneStep 更优。

一个需要注意的小点：当前 `metrics.csv` 里的 `max_single_vehicle_deceleration` 是所有车辆的最小加速度，包含 MV。OneStep 的 `-2.265` 主要来自 MV 自身轨迹，不代表主线急减速。主线扰动分析应优先看 `mainline_acceleration_energy`、`affected_mainline_vehicle_count`、`mainline_speed_recovery_time` 和每辆主线车的聚合表。
